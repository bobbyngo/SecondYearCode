This section will cover the basics of parameterization, parameterized types, generic types and generics in Java.

~Task 4.2~

As an aside, please note that the information on generics provided in this lab is relatively narrow. There are more ways to incorporate generics and parameterization in Java, such as wildcards, multi-generics, bounded generics, and generic methods (<- wow!). These are advanced concepts and should be considered beyond the scope of the course. However, you should understand roughly what is happening when you instantiate a parameterized type like ArrayList.

You are going to write a simple generic class now. The file (Block.java) is provided and started for you.

Complete Block.java such that the errors in Main.java go away, and the correct output is produced (you get "Correct" when pressing the Check button).

If it is not clear from the code, you need to implement methods (and a constructor) accordingly in the Block class. In addition to the shown methods, you need a getHash() method to return the Block's hash.

A brief explanation of some strange things you likely noticed in Main.java and Block.java:

  • <T> : Remember, this is just a placeholder. We can call our parameter type anything. But by convention we use one of E, T, K, V. We have seen E in ArrayList, which denotes we have a parameterized element type (collection). T denotes an object type. Our class is holding a single object of variable type, so we use T.
  • <?> : This is a wildcard type! We are saying that we do not know what type to expect. Why do we need it? We need it because a Block can hold any type of object, and there is no way for us to easily determine what the next Block is parameterized to. Note: you will need to use it in your solution to Block.java.
  • 2004L : If you are curious, this is just a long literal.
